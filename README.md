# NPCLife

**LLM 驱动的游戏叙事中间件**

NPCLife 是一个可嵌入游戏引擎的 C# 库。游戏世界产生的事件会自动触发 AI 驱动的剧情生成，为 NPC 提供有因果关系的动态叙事。

---

## 它能做什么

### 自动剧情生成

游戏世界中的事件（战斗、对话、探索、环境变化……）被上报到 NPCLife。当积累到一定阈值时，AI 会自动审查这些事件，决定哪些值得发展为剧情线，并生成 NPC 台词和剧情推进。

```
游戏事件 → 事件池积累 → AI 审查 → 剧情线生成 → NPC 台词输出
```

### 剧情线分支与合并

每条剧情线是一个独立的**工作空间（Workspace）**，支持：

- **Branch（分支）**：从主线分叉出子线，子线继承父线的历史
- **Merge（合并）**：将多条子线合并回主线，保留各自的叙事历史
- **独立生命周期**：每条线可以独立挂起、恢复、完成或废弃

```
主线: "殖民地危机"
  ├── 分支: "Alpha 的背叛"
  ├── 分支: "Beta 的救援"
  └── 合并: "两线汇合" → 主线
```

### 多角色协作

NPCLife 内置三种 AI 角色，各司其职：

| 角色 | 职责 |
|------|------|
| **Director（导演）** | 审查事件池，决定哪些事件值得发展成剧情线，将事件路由到对应的工作空间 |
| **Screenwriter（编剧）** | 为分配到的剧情工作空间生成具体的 NPC 台词和叙事 |
| **Freelancer（临时编剧）** | 处理一次性或临时性事件 |

角色之间通过**异步消息传递**协作：导演将事件路由给编剧时可附加留言和关键词；编剧完成一轮叙事后可向导演留言说明当前状态（继续推进 / 需要新事件 / 可关闭）。导演在下次轮询时读取这些留言，调整后续的事件分配策略。

### 上下文隔离

每个工作空间维护独立的事件池、对话历史和角色集合。不同剧情线之间互不干扰，避免上下文污染。

### 事件阈值触发

AI 不会每帧都调用 LLM。事件在池中积累，达到**数量阈值**或**累计重要度阈值**时才触发 AI 处理。这控制了 API 调用频率和成本。

## 工作流程

1. **游戏上报事件**：游戏循环中，将发生的事件（战斗、对话等）推送到事件池
2. **阈值检测**：事件池自动检测是否达到触发阈值
3. **导演激活**：阈值达到后，导演 AI 审查事件列表，决定路由策略
4. **事件路由**：导演将事件分配到对应的工作空间（没有合适的则新建）
5. **编剧生成**：工作空间内的事件积累触发编剧 AI，生成具体台词和叙事
6. **台词输出**：编剧通过工具调用逐句推送台词，游戏侧立即显示
7. **轮次归档**：编剧完成一轮后归档前情提要和剧情结果，向导演留言
8. **循环继续**：导演下次轮询时读取留言，调整策略

## 集成要求

NPCLife 是纯逻辑库，不依赖任何游戏引擎。宿主游戏需要提供：

- **存储适配器**（`IStorage`）：用于持久化工作空间状态
- **日志适配器**（`ILogger`）：用于框架日志输出
- **LLM 服务**（`ILlmService`）：对接具体的 LLM API（OpenAI / Anthropic / 自建）
- **时间提供者**：返回当前时间字符串，用于时间戳记录

游戏侧还可以注入自定义的 MCP 工具（角色查询、环境感知、关系网络等），供 AI 在生成叙事时调用。

---

## 适用场景

- 需要为 NPC 生成动态剧情线（而非固定脚本）的游戏
- 希望剧情能响应玩家行为和环境变化的游戏
- 开发者能够注入自己的存储、日志、LLM 服务实现
